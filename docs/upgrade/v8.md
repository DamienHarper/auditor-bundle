# Upgrading to 8.0

> **Migrate from auditor-bundle 7.x to 8.0**

This guide covers upgrading from auditor-bundle 7.x to 8.0.

## вљ пёЏ DoctrineProvider extraction completed in auditor 5.0

`auditor-bundle` 8.0 targets `damienharper/auditor` 5.x, in which the embedded
`DH\Auditor\Provider\Doctrine` namespace has been **fully removed** from the core library.

All Doctrine ORM functionality now lives exclusively in
[`damienharper/auditor-doctrine-provider`](https://github.com/DamienHarper/auditor-doctrine-provider),
which auditor-bundle already requires as a direct dependency.

> [!IMPORTANT]
> The class namespace **has not changed**. Classes are still under `DH\Auditor\Provider\Doctrine\*`
> вЂ” they simply come from the `auditor-doctrine-provider` package instead of the `auditor` core.
> No import statements need to be updated.

### For auditor-bundle users

No action required. The bundle automatically requires `damienharper/auditor-doctrine-provider`
and wires everything transparently. Your `dh_auditor.yaml` configuration continues to work
without any changes.

### For users using DoctrineProvider directly (without the bundle)

If you use `DoctrineProvider` directly in a non-bundle project, install the standalone package
explicitly:

```bash
composer require damienharper/auditor-doctrine-provider:^2.0
```

All imports remain identical:

```php
// Works in both 4.x (from auditor core) and 5.x (from auditor-doctrine-provider)
use DH\Auditor\Provider\Doctrine\DoctrineProvider;
use DH\Auditor\Provider\Doctrine\Configuration;
use DH\Auditor\Provider\Doctrine\Persistence\Reader\Reader;
// ...
```

### `NeedsConversionToAuditableType` (if applicable)

If you implemented the `NeedsConversionToAuditableType` interface on a custom Doctrine type,
update the import:

```php
// Before (auditor 4.x)
use DH\Auditor\Transaction\NeedsConversionToAuditableType;

// After (auditor 5.x / auditor-doctrine-provider 2.x)
use DH\Auditor\Provider\Doctrine\Transaction\NeedsConversionToAuditableType;
```

## вљ пёЏ Audit Table Schema Reform (schema version 2)

auditor-bundle 8.0 also ships `damienharper/auditor-doctrine-provider` 2.x, which introduces a
**revised audit table schema**. The new schema:

- Replaces four separate `blame_*` columns with a single `blame JSON` column
- Replaces the SHA-1 `transaction_hash VARCHAR(40)` with a ULID `transaction_id CHAR(26)`
- Adds a `schema_version` column to distinguish legacy and current diffs formats
- Adds two composite indexes: `(object_id, type)` and `(blame_id, created_at)`
- Unifies the `diffs` JSON format across all operation types

Existing data is **not modified automatically** вЂ” run `audit:schema:migrate` after upgrading.

### Migrating Existing Audit Tables

```bash
# Preview the SQL
bin/console audit:schema:migrate --dump-sql

# Run the migration
bin/console audit:schema:migrate --force

# Optionally convert legacy diffs JSON to the v2 format (safe to skip)
bin/console audit:schema:migrate --force --convert-diffs
```

### Breaking Changes

**`$entry->transactionHash` renamed to `$entry->transactionId`**

```php
// Before (7.x)
$entry->transactionHash;

// After (8.0)
$entry->transactionId;
```

The old property is kept as a deprecated alias.

**Blame properties now read from `blame` JSON column**

`$username`, `$userFqdn`, `$userFirewall`, and `$ip` still work вЂ” they now read from the unified
`blame` JSON column. No code changes required.

**Remove entries now include a full entity snapshot**

For schema version 2 entries, `getDiffs()` on a `remove` entry returns the full entity state
before deletion, with `old` populated and `new` set to `null` for every field.

**`getDiffSource()` available on all entry types**

Use `getDiffSource()` to access the entity metadata (`id`, `class`, `label`, `table`) that is
now present on all operation types in schema version 2.
