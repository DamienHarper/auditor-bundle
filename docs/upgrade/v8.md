# Upgrading to 8.0

> **Migrate from auditor-bundle 7.x to 8.0**

This guide covers upgrading from auditor-bundle 7.x to 8.0.

## ⚠️ DoctrineProvider extraction

`auditor-bundle` 8.0 introduces [`damienharper/auditor-doctrine-provider`](https://github.com/DamienHarper/auditor-doctrine-provider) as a **direct dependency**.

The `DoctrineProvider` and all related classes that were previously bundled inside `damienharper/auditor` (under the `DH\Auditor\Provider\Doctrine\` namespace) are now **deprecated** in `damienharper/auditor` 4.x and will be **removed in `damienharper/auditor` v5.0**.

### For auditor-bundle users

No action required. The bundle automatically requires `damienharper/auditor-doctrine-provider` and wires everything transparently. Your `dh_auditor.yaml` configuration continues to work without any changes.

### For users using DoctrineProvider directly (without the bundle)

If you use `DoctrineProvider` directly in a non-bundle project, install the standalone package explicitly:

```bash
composer require damienharper/auditor-doctrine-provider
```

Update any import statements if you relied on the old bundled location:

```php
// Before (auditor 4.x, deprecated)
use DH\Auditor\Provider\Doctrine\DoctrineProvider;

// After (auditor-doctrine-provider)
use DH\AuditorDoctrineProvider\DoctrineProvider;
```

> [!NOTE]
> The old `DH\Auditor\Provider\Doctrine\` classes still exist in `damienharper/auditor` 4.x as deprecated aliases. They will be removed in v5.0.

## 🚀 Upgrade Steps

### 1️⃣ Update auditor-bundle

```bash
composer require damienharper/auditor-bundle:^8.0
```

This will automatically pull in `damienharper/auditor-doctrine-provider`.

### 2️⃣ Clear Cache

```bash
bin/console cache:clear
```

### 3️⃣ Verify Schema

```bash
bin/console audit:schema:update --dump-sql
```
